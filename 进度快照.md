## 产品定位(P7 后)
不是"SQL 生成器",而是**对话式数据分析 Copilot**:面向**不会写SQL的业务用户**(运营/市场/产品),问一句话→拿答案+洞察+图表,SQL是可选审计。前6阶段=引擎(评测/检索/自纠错/可靠/可观测),P7起=产品层。

## 关键事实
- 语言/工具:Python 3.14 + **uv**
- 大模型:**DeepSeek**(OpenAI 兼容),配置在 `.env`(已 gitignore)
- 数据:Spider dev 集在 `data/spider/`(dev.json 1034 题 + 20 个 sqlite 库);本地样例库 `data/sample.sqlite`
- **关键洞察**:执行错误率 0% → 瓶颈是"逻辑答错"非"语法错" → 优化重点是检索/few-shot,不是单纯自纠错
- **Phase 3 检索**:词法检索器召回率已 100%(到顶),嵌入版已写好可插拔但**没必要烧钱跑**;`--schema {full,lexical,embed}` 切换;FK 扩展保 JOIN;表召回率用 sqlglot 解析 gold SQL 度量
- **Phase 4 自纠错**:`generate_with_correction()` 行动→观察报错→改→重试(有界 max_retries);`--correct N`(eval+ask.py);Spider 0 触发;正确性靠单测(假LLM)证明;**待办:few-shot 示例检索才是真正撬准确率的杠杆**
- **Phase 4B few-shot**:`examples.py` 倒排索引检索 train 池的 Q→SQL;`--fewshot K`;**核心:train 池含全部 dev 库→检索时强制排除同库防泄漏(单测锁死)**;65→77%;三开关 `--schema/--correct/--fewshot` 正交可叠加做消融
- **Phase 5 可靠性+可观测性**:`tracing.py`(每次LLM调用落 `data/traces/llm-YYYYMMDD.jsonl`)+ `scripts/trace_report.py`(按模型出调用/成功率/token/成本/p50p95);Langfuse 配 key 可选;`guard.py` sqlglot 拦危险SQL(execute_sql 内调用);`db.py` 看门狗超时;`llm.py` 主备降级(`_call` 是注入点);单价进 config,`pricing.py` 共用
- **Phase 6 API+前端**:`service.py`(编排层,`answer_query`/`list_databases`/`suggest_chart`)+ `api.py`(FastAPI:`/` `/api/databases` `/api/ask` `/api/traces/summary`)+ `web/index.html`(定制单页,Chart.js,表格/图表切换,顶部实时成本)+ `scripts/serve.py`;启动:`uv run python scripts/serve.py` → http://127.0.0.1:8000
